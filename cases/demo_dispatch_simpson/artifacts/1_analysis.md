## 0. 变量角色（先把"谁是谁"钉死，后面全靠它）

| 变量 | 角色 | 取值 | 说明 |
|---|---|---|---|
| `policy` | **处理变量 / treatment** | A / B | 我们想评估其效果的"干预" |
| `on_time` | **结果变量 / outcome** | 1 是 / 0 否 | 按时到达与否，二值 |
| `severity` | **疑似混杂 / confounder** | Low / High | 既可能影响"用哪套策略"，又直接影响"是否按时"——典型混杂结构 |
| `call_id` | 标识符 | 1..700 | 仅行号，**不入模** |

> 这是全案核心判断：`severity` 同时关联处理与结果 → 不控制它，合并率会骗人。是否真混杂**需 Scout 用数据实证**（下文陷阱区）。

## 1. 子问题清单（Q1/Q2/Q3）

| 编号 | 子问题 | 必答/加分 | 交付物（口径） |
|---|---|---|---|
| **Q1** | 哪套策略"按时到达率"更高？推荐哪一套？ | **必答** | 给出 A、B 各自的合并按时率 + 一个推荐 |
| **Q2** | 这个结论可靠吗？是否存在混杂导致的误判？ | **必答** | 实证判断：**按 `severity` 分层后率是否反转**；若反转，说明合并率被污染 |
| **Q3** | 公平比较两套策略的正确做法是什么？给可落地建议 | **必答** | 给出公平比较的方法（分层/标准化/回归调整）+ 对市政的落地建议 |
| Q3+ | 量化"公平比较后"的策略差异并给不确定性 | **加分** | 标准化率之差、或 logistic 系数 + 置信区间（题面未强制要求"区间"，故列加分） |
| Q2+ | 解释混杂的**机制**（为何 A/B 在严重度上分布不均） | **加分** | 用数据描述两策略在 Low/High 上的派遣占比差异，讲清"为什么会反转" |

> 三问是**层层递进的自纠结构**：描述（Q1）→ 质疑（Q2）→ 修正（Q3）。Q1 的"直觉答案"很可能与 Q3 的"正确答案"相反——这正是本题考点，**不是 bug**。

## 2. 关键定义表（题目模糊处，先给可操作定义）

| 术语 | 可操作定义 | 来源 |
|---|---|---|
| **按时到达率（合并率）** | 某策略下 `on_time==1` 的行数 ÷ 该策略总行数。即 `mean(on_time \| policy)` | 题面原意（"按时到达率"）+ Analyst 操作化 |
| **分层率 / 条件率** | 在固定 `severity` 后再算的按时率，即 `mean(on_time \| policy, severity)` | **Analyst 假设**：Q2"是否混杂"的标准检验口径就是分层；题面未明说"分层"二字 |
| **"更优 / 更高"** | 默认指**按时到达率更高**。但 Q2/Q3 暗示真正的"更优"应是**剔除混杂后**仍更高的策略，而非合并率更高者 | 题面"更高"为字面；"剔除混杂后更优"为 Analyst 推断的出题真意，**需人确认** |
| **混杂（confounding）** | 存在第三变量（此处 `severity`）同时影响处理选择与结果，使处理-结果的边际关联≠因果效应 | 标准统计定义 |
| **Simpson 悖论 / 率反转** | 合并数据里 A>B，但每个 `severity` 子层内都 B>A（或反之）。**本案核心嫌疑** | Analyst 标注：目录名暗示，但**必须由数据证实，不能预设** |
| **公平比较** | 让两策略在**相同的严重度构成**下比较，常见三法：①分层比较 ②标准化率（standardization，按统一权重加权各层率）③回归调整（logistic：on_time ~ policy + severity） | Analyst 操作化（题面只说"公平比较"未给方法） |
| **目标时限** | 题面未给具体秒/分阈值；`on_time` 已是算好的 0/1，**无需我们定义阈值** | 题面：`on_time` 直接给出 |

## 3. 建模假设清单（每条含"为什么 + 可能错在哪"）

| # | 假设 | 为什么这么假设 | 可能错在哪（代价） |
|---|---|---|---|
| A1 | `severity` 是**唯一**相关混杂，控制它即可公平比较 | 数据只有这三个字段，无其他协变量可控 | 现实里还可能有时段/区域/路况/警力等**未观测混杂**——本数据测不到，结论只能在"给定可观测变量"下成立。Critic 须把"无未观测混杂"写成显式前提，**不可声称证明了因果** |
| A2 | 每行是一次**独立**出警，可直接计数算率 | 表格为记录型、无重复 ID 提示 | 若同一辖区/同一事件被多次记录、或存在时间自相关，独立性破裂，置信区间会偏窄。需 Scout 核 `call_id` 是否唯一 |
| A3 | `on_time` 的口径在 A、B 间一致（同一"目标时限"标准） | 题面把 `on_time` 当统一结果字段 | 若两策略各自定义了不同时限，"按时率"不可比，整题失效。**需核实**口径是否同一 |
| A4 | 样本对该市出警总体有代表性，可外推到市政决策 | 题面要"向市政推荐" | 若是试点期/特定区域选择性记录，存在**选择偏差**，外推受限。建议加分项里注明适用范围 |
| A5 | `severity` 的 Low/High 划分客观且在两策略间含义一致 | 作为分层依据必须可比 | 若 B 策略系统性把同类呼叫判得更重/更轻，分层本身就被污染（"分层变量内生"），标准化也救不回。Critic 留意 |
| A6 | 二值结果用率/卡方/logistic 即可，无需复杂时序模型 | 无时间戳、无机理过程 | 若实际关心"响应时间分布"而非二值率，会丢信息——但数据只给了 0/1，只能做到这一步 |

## 4. 已知陷阱清单（交 Critic 重点盯）

| 陷阱 | 具体风险 | 谁盯 / 怎么防 |
|---|---|---|
| **C1 · Simpson 式率反转（头号）** | 合并率 A>B，但 Low 层、High 层内都可能 B>A。**若直接用合并率下结论 = 读错题、头号失分** | Scout 必须先出**分层交叉表**实证；Writer **严禁**只用合并率收尾。Router 铁律已锁此条 |
| **C2 · 选择偏差 / 派遣不随机** | 两策略大概率**没被随机分配**到呼叫上（如 A 多接 Low、B 多接 High）。这正是混杂产生的源头 | Scout 出"policy × severity"占比表，量化失衡程度；若失衡严重，合并率毫无意义 |
| **C3 · 把"相关"当"因果"** | 即便分层后 B 更优，也只是"在可观测严重度下"的关联，未必是 A2 之外其他混杂被控住 | Critic 强制 Writer 用"在给定 severity 下"的限定措辞，不写"B 导致更高按时率" |
| **C4 · 样本量/随机波动** | 700 行分到 2 策略 × 2 层 = 4 格，某格若样本太少，分层率不稳，反转可能是噪声 | Solver 给每格 n 与置信区间；Critic 查最小格样本量，触发回退判断 |
| **C5 · 分层变量内生（A5 延伸）** | 若 severity 判定本身受 policy 影响，分层/标准化都失效 | Critic 核 severity 定义是否独立于 policy |
| **C6 · 标准化权重选择** | 标准化率随"用哪套人群权重"（A 的、B 的、还是总体）变化，结论可能随权重漂移 | Modeler 明确权重来源并做敏感性；CP2 让人拍板方法 |
| **C7 · 过度解读 / 外推越界** | 把试点样本结论当全市铁律（A4） | Writer 注明适用范围，Judge 复盘把关 |

> 给 Scout 的一句话指令：**先别下结论，先把 `policy×severity×on_time` 的 2×2×2 交叉表和各层率摆出来**——反转是否发生、哪格样本最小，一表见分晓。
