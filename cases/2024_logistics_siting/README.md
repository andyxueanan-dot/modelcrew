# 优化案例 · 应急服务站选址（连续 p-中位问题）

> **ModelCrew "换题型即用" 的硬验证**：前两个 demo 都是**数据型(C)**；本案例是**优化型(B)**。
> 同一支团队、**定义一行不改**，由 Router 自动判为优化型、走另一条召唤链路跑通。
> 本案例由**已移植并验证的 Claude Code ModelCrew subagents**（见 `.claude/agents/`）运行。

## 业务背景
为一座城市的 64 个社区规划 **K=4 个应急服务站**（消防/急救/物资点）的位置，
目标：最小化**居民到最近站点的人口加权总距离**（越小=应急响应越快、越公平）。
这是经典的**连续加权 p-中位问题（multi-source Weber / continuous p-median）**——**非凸**，有大量局部最优。

## 数据
- `data/communities.csv`：64 个社区，字段 `id, x, y, population`（坐标在 0–100km 城区内，由固定种子 `seed=42` 预生成的 5 个居住组团）。
- 总人口 346,573。

## 核心子问题
- **Q1**：4 个服务站最优放在哪？最小总加权距离是多少？
- **Q2**：求得的解**真的是最优**吗？（非凸优化最容易"把局部最优当全局最优"）
- **Q3**：K=4 合理吗？（站点数 vs 成本的敏感性/拐点）
- **Q4**：这个模型直接用于真实应急规划，有什么风险？（欧氏距离 vs 真实路网、容量/覆盖半径、需求静态假设）

## 为什么是反幻觉 Critic 的完美舞台（新维度）
前两个数据型案例考的是 Critic 的"随机性/过拟合/公平"；本优化案例专考它的 **「可行性 / 最优性」** 维度：
1. **局部最优陷阱**：单次局部搜索会**自信地**报出一个"最优"布局，其实只是众多局部最优之一；
2. **可行性**：站点是否落在城市边界、是否满足约束；
3. **最优性证书**：启发式（多起点）找到的"最优"也**没有全局最优证明**，措辞要诚实；
4. **K 的选择**：成本随 K 单调下降，"最优 K"是预算/边际权衡而非纯优化结论。

## 运行路线（Router 判为"优化型"）
Analyst → Modeler(p-中位 / 加权 Lloyd + 多起点) → Solver(求解+敏感性+可视化) → ⭐Critic(可行性/**局部vs全局最优**/K 选择) → Writer。
