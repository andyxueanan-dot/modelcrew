# 金融案例 · 信用卡违约预测（信用风险评分）

> **ModelCrew 可复用性的真实验证**：同一支 AI 团队，从网球题（体育）换到信用风险题（金融），
> 团队定义一行不改，只新建本案例文件夹 + 换数据，即可重跑全流程。
> 面向**银行风控 / 模型治理(model governance)** 场景——对口金融科技岗。

## 业务背景
银行/信用卡机构需要在放贷前评估客户**下月是否会违约**，以控制信用风险。
本案例用真实公开数据训练一个违约预测模型，并**重点用反幻觉 Critic 把它当成"上线前的模型评审"来审**——
这正是金融机构最看重的"模型可靠性 + 合规公平"。

## 数据
- 来源：UCI「default of credit card clients」（台湾信用卡，2005 年 4–9 月）
- 规模：**30,000 条 × 24 特征**，标签 `default.payment.next.month`（1=下月违约）
- 特征：额度 LIMIT_BAL、人口属性 SEX/EDUCATION/MARRIAGE/AGE、近 6 月还款状态 PAY_0..6、账单 BILL_AMT1..6、还款额 PAY_AMT1..6
- **关键事实**：违约率 **22.1%**（类别不平衡）；"全部预测为不违约"也能拿到 **77.9% 准确率**

## 核心子问题
- **Q1**：能否用客户特征预测下月违约？
- **Q2**：哪些因素对违约最重要？（如近期还款状态 PAY_0）
- **Q3**：在 22% 这种**类别不平衡**下，如何**正确**评估模型？（准确率会骗人，要看 AUC/KS/召回/精确率）
- **Q4**：把模型用于真实风控，有什么**合规与公平**风险？（用性别/年龄做特征 → 歧视/合规隐患）

## 为什么是反幻觉 Critic 的完美舞台
信用风险建模有几个最容易"看起来很好、其实有问题"的坑，正好让 Critic 当"上线评审委员会"：
1. **准确率陷阱**：不平衡数据下高准确率毫无意义（77.9% baseline）；
2. **数据泄漏**：是否误用了未来信息 / 不该用的字段；
3. **公平与合规**：SEX/AGE/EDUCATION 作为特征，在真实银行场景涉嫌歧视、违反公平信贷；
4. **过拟合 / 阈值 / 校准**：风控更关心高风险客户的召回与决策阈值，而非整体准确率。

## 运行路线（Router 判为"数据型/分类"）
Analyst → Scout(重，类别不平衡+敏感属性) → Modeler(逻辑回归 baseline) → Solver(AUC/KS/PR + 特征重要性) → ⭐Critic(准确率陷阱/泄漏/公平) → Writer(信用风险建模报告)。
