## 审计焦点 1：准确率陷阱 ⭐

### 问题描述
违约率 22.1%，"全部预测不违约"的 naive baseline 准确率 = 77.9%。如果以 accuracy 为唯一指标，一个**完全不做任何预测的模型**就能拿到 77.9%。

### 审计发现

| 模型 | Accuracy | vs Naive Baseline | 有意义吗？ |
|------|----------|-------------------|-----------|
| Naive (全预测 0) | 0.779 | — | 零信息 |
| Logistic Regression | 0.694 | **−8.5pp** ❗ | LR 的 accuracy 反而**低于**naive baseline |
| XGBoost | 0.760 | −1.9pp | 仍低于 naive baseline |
| Random Forest | 0.782 | +0.3pp | 仅比 naive 高 0.3pp |

**关键发现**：三个模型中，**两个的 accuracy 低于"什么都不做"的 baseline**。这完美诠释了为什么 accuracy 在不平衡数据上是欺骗性指标。

### 正确的评估方式

| 指标 | Naive Baseline | XGBoost | 解读 |
|------|---------------|---------|------|
| Accuracy | 0.779 | 0.760 | Accuracy 说 XGBoost 比 naive 差 |
| **AUC-ROC** | 0.500 | **0.782** | AUC 说 XGBoost 有真实区分力 |
| **KS** | 0.000 | **0.422** | KS 说 XGBoost 能有效分离好坏客户 |
| **Recall** | 0.000 | **0.616** | XGBoost 能捕获 62% 的违约客户 |
| **F1** | 0.000 | **0.532** | 综合考虑 precision 和 recall |

### 裁决
**⚠️ 警告**：任何以 accuracy 作为主要 KPI 的报告都是误导性的。Modeler 和 Solver 正确地使用了 AUC/KS/F1/PR-AUC 进行评估——这一点值得肯定。但**论文中如果提到 accuracy，必须同时说明 naive baseline 和准确率陷阱**。

## 审计焦点 2：数据泄漏 ⭐

### 问题描述
PAY_0 是"最近一期还款状态"。在预测"下月是否违约"时，PAY_0 是否包含了不应该知道的未来信息？

### PAY_0 的时间线分析

```
时间线:  ...  →  4月  →  5月  →  6月  →  7月  →  8月  →  9月
                    ↑                       ↑
              PAY_6 (6月前)           PAY_0 (最近月)
                                          │
                                     default.payment.next.month
                                     (下月 = 10月？)
```

**关键问题**：
- PAY_0 编码的是"最近一个月"的状态。如果数据在 9 月收集，PAY_0 = 9 月还款状态，target = 10 月是否违约 → **不构成泄漏**（9 月信息在预测 10 月时是已知的）
- 但如果 PAY_0 编码的是"当月"状态，且与 target 存在时间重叠 → **可能构成泄漏**

### PAY_0 影响力的异常信号

| 证据 | 含义 |
|------|------|
| PAY_0 占 XGBoost gain 的 **37.1%** | 单一特征占比过高，可能是泄漏的信号 |
| PAY_0 与 target 的相关系数 = **0.325** | 远高于第二特征 PAY_2 (0.264) |
| PAY_0=2 (逾期 2 月) 的客户，违约率应接近 100% | 如果已逾期 2 月，"下月违约"几乎是确定的——这可能不是预测，而是**同义反复** |

### 敏感性测试：移除 PAY_0

如果 PAY_0 存在泄漏嫌疑，移除它后模型表现如何？这是一个关键的 sanity check。

**建议**：在论文中报告 "without PAY_0" 的 AUC 作为 sensitivity analysis。如果 AUC 大幅下降（如从 0.78 降至 0.72），说明模型过度依赖 PAY_0，其预测力的相当部分来自潜在的泄漏或同义反复。

### 裁决
**⚠️ 存疑**：PAY_0 的时间锚点需要明确记录。当前文档（UCI dataset）未给出精确的时间定义。模型对 PAY_0 的过度依赖（37%）是一个风险信号。建议在论文中将 PAY_0 的影响明确标注为"可能包含近期财务困境的强信号"，并报告移除 PAY_0 后的 sensitivity analysis。

## 审计焦点 3：公平与合规 ⭐⭐

### 问题描述
模型使用了 SEX(性别)、AGE(年龄)、EDUCATION(教育程度)、MARRIAGE(婚姻状况) 作为特征。在真实银行场景中，这些是受保护属性(protected attributes)，将信贷决策与这些属性挂钩可能违反公平信贷法规。

### 法规框架

| 法规 | 受保护属性 | 适用性 |
|------|-----------|--------|
| **美国 ECOA** (Equal Credit Opportunity Act) | Race, Color, Religion, National origin, Sex, Marital status, Age | SEX, MARRIAGE, AGE 直接受限 |
| **欧盟 GDPR + AI Act** | 特殊类别数据 + 高风险 AI 系统 | 信用评分被归类为高风险 |
| **台湾个人资料保护法** | 类似保护 | 数据集来源地 |

### 差异影响分析 (Disparate Impact)

| 属性 | 组别 | 违约率 | 差异 | 法律风险 |
|------|------|--------|------|---------|
| **SEX** | Male | 24.2% | +3.4pp | ⚠️ 如果模型对男性更频繁拒绝信贷 |
| | Female | 20.8% | | |
| **AGE** | 61–79 | 26.8% | +6.4pp vs 31–40 | ⚠️ 年龄歧视 |
| | 31–40 | 20.4% | | |
| **EDUCATION** | High School | 25.2% | +6.0pp vs Graduate | ⚠️ 教育歧视 |
| | Graduate | 19.2% | | |

### 关键发现：移除敏感属性不影响模型

| 模型 | AUC | KS |
|------|-----|-----|
| XGBoost (含敏感属性) | 0.7815 | 0.4223 |
| XGBoost (不含敏感属性) | 0.7789 | 0.4234 |
| **Δ** | **−0.003** | **+0.001** |

**AUC 仅下降 0.003（0.3 个百分点）**——这是一个统计上可忽略的差异。这意味着：
1. 敏感属性对模型预测力几乎没有贡献
2. 可以安全地移除它们以满足合规要求
3. 保留它们没有任何业务理由，只有法律风险

### 整改建议

1. **立即移除** SEX、AGE、EDUCATION、MARRIAGE 四个特征 → ΔAUC < 0.3%，零性能损失
2. **代理变量审计**：检查 LIMIT_BAL 和 BILL_AMT 是否作为 AGE/EDUCATION 的代理变量（proxy）——高学历→高收入→高额度，间接引入歧视
3. **公平性指标监控**：部署后按受保护群体跟踪 approval rate 和 default rate 差异
4. **文档化**：在模型卡(Model Card)中记录敏感属性审计结果和移除理由

### 裁决
**❌ 模型不可部署**（在移除敏感属性之前）。当前模型使用了受保护属性，在真实银行场景下：
- 违反 ECOA 的 sex/marital status/age 保护条款
- 可能引发歧视诉讼
- 监管审查中模型将无法通过验证(Validation)

**整改路径明确且成本低**：移除 4 个特征，AUC 损失 0.003。

## 审计焦点 4：过拟合、阈值与校准

### 4a. 过拟合检查

| 信号 | 评估 |
|------|------|
| XGBoost early stopping = 43/300 iterations | ✅ 远未达到 max，未过拟合 |
| Train/test AUC gap | 未报告训练集 AUC——**建议补充** |
| RF vs XGBoost AUC: 0.779 vs 0.782 | ✅ 接近，说明模型稳定 |
| LR vs XGBoost AUC: 0.717 vs 0.782 | ✅ 非线性模型确实更好，但不是巨大跳跃 |

**过拟合风险**：低到中等。Early stopping 有效防止了过拟合，但缺少 train AUC 对比。

### 4b. 决策阈值分析

当前使用 threshold = 0.5，但银行风控通常关注的是：
- **高风险客户**：threshold 应更低（如 0.3），宁可误判也要多拦截
- **KS 最优阈值**：KS 曲线显示最优阈值约为 0.30–0.35

| Threshold | Recall | Precision | 业务含义 |
|-----------|--------|-----------|---------|
| 0.50 | 0.616 | 0.468 | 当前设定：漏检 38% 违约 |
| 0.35 (est.) | ~0.75 | ~0.38 | 更高召回，更多误判 |
| 0.20 (est.) | ~0.85 | ~0.30 | 激进风控：捕获 85% 但误判率高 |

**建议**：报告多个阈值下的 precision-recall 权衡表，让业务方选择。不要硬编码 0.5。

### 4c. 概率校准

模型输出的 P(default) 是否可信？例如，如果模型给某客户 P=0.70，那么这类客户中是否确实约 70% 会违约？

- XGBoost 的概率输出**通常未经校准**——它是为 AUC 优化的，不是为概率校准优化的
- 银行风控决策（如定价、额度调整）依赖准确的违约概率 → **需要 Platt scaling 或 isotonic regression 校准**
- 当前 Solver 未做校准分析——**这是论文的一个局限性，需要明确说明**

### 裁决

| 方面 | 裁决 | 理由 |
|------|------|------|
| 过拟合 | ✅ 低风险 | Early stopping 有效，模型间一致性良好 |
| 阈值选择 | ⚠️ 不足 | 只报告了 threshold=0.5 的结果，缺少多阈值分析 |
| 概率校准 | ⚠️ 未做 | XGBoost 概率未校准，不适合直接用于定价决策 |

## 7 项标准审计

### 1. 随机性 / 显著性
模型评估使用了标准的 train/test split（70/30 分层）。9,000 条测试样本足够支撑 AUC 的置信区间（AUC=0.782, 95% CI ≈ [0.771, 0.793]）。✅ 充分。

### 2. 相关 ≠ 因果
PAY_0 与违约的高相关性（r=0.325）可能反映因果关系（逾期还款→更可能违约），也可能反映共同原因（财务困境→逾期+违约）。论文应区分"预测"和"因果"。⚠️ 需注意措辞。

### 3. 过拟合 / 泄漏
- 过拟合：低风险（early stopping, 模型一致性）✅
- 泄漏：PAY_0 的时间锚点存疑 ⚠️（见焦点 2）
- ID 已正确排除 ✅

### 4. 量纲 / 边界
- LIMIT_BAL (TWD) 和 BILL_AMT (TWD) 单位一致 ✅
- PAY_0 编码 -2 到 8，含义明确 ✅
- 利用率 (BILL_AMT/LIMIT_BAL) 可超过 1.0（超额消费）→ 合理 ✅

### 5. 可行性 / 最优性
- XGBoost 推理延迟 <1ms/样本，部署可行 ✅
- KS=0.422 > 0.30 行业标准 ✅
- 但 Recall=0.616 意味着 38% 违约漏检——业务可接受性取决于损失成本矩阵 ⚠️

### 6. 样本 / 外推
- 2005 年台湾数据 → 2024 年部署 = 19 年概念漂移 ❌
- 仅一家银行的数据 → 推广到其他银行/国家需谨慎 ⚠️

### 7. 假设回溯
- A2 (PAY_0 时间锚点): 未完全验证 ⚠️
- A3 (敏感属性可用): 被合规审计否决 ❌
- A6 (i.i.d. + 无漂移): 19 年差距下不成立 ❌

## 结论裁决详述

### C1: XGBoost 在 AUC 上优于 LR → ✅ 成立
AUC 差距 0.065（0.782 vs 0.717），在 9,000 样本的测试集上高度显著。这是稳健的发现。

### C2: PAY_0 是最重要的预测变量 → ⚠️ 存疑
PAY_0 确实贡献了 37% 的 XGBoost gain 和最高的 LR 系数。但 37% 的集中度本身是一个风险信号——可能反映了 PAY_0 作为"近违约"指标的泄漏性质。论文应报告移除 PAY_0 后的 sensitivity analysis。

### C3: 模型 AUC > 0.75 → ⚠️ 存疑
AUC = 0.782 确实 > 0.75，KS = 0.422 > 0.30。但：
- Recall = 0.616 意味着 38% 违约客户漏检
- PR-AUC = 0.554 远低于 ROC-AUC（类别不平衡下的典型表现）
- 对于银行风控而言，0.782 的 AUC 是"可接受"但不是"优秀"

### C4: 信用利用率和延迟趋势是重要次要特征 → ✅ 成立
utilization 和 pay_delay_trend 在 LR 和 XGBoost 中均有显著贡献，且系数方向符合业务直觉（额度利用率升高 + 延迟恶化 → 违约风险增加）。

### C5: 模型可安全部署 → ❌ 被否决
四项阻断性问题：
1. 使用受保护属性（SEX/AGE/EDUCATION/MARRIAGE），违反公平信贷法规
2. 概率未校准，不适合用于风险定价
3. 2005 年数据训练，19 年概念漂移风险
4. 仅单一银行数据，泛化能力未验证

### C6: 移除敏感属性不损害模型 → ✅ 成立
ΔAUC = 0.003，ΔKS = +0.001。统计和业务上均可忽略。**这是支持 C5 整改的关键证据**：移除敏感属性的成本极低。

## 对论文写手的指令

1. **可以自信地写**：C1（XGBoost 优于 LR）、C4（次要特征的业务合理性）、C6（移除敏感属性零损失）
2. **必须带 caveat 写**：C2（PAY_0 的重要性 + 泄漏风险）、C3（AUC 达标但 Recall 不足）
3. **不可作为正面结论写**：C5（模型可部署）。应改为"模型展示了有希望的预测力，但在合规、校准、时效性方面存在阻断性限制"
4. **必须包含的章节**：
   - 准确率陷阱的明确讨论（与 naive baseline 对比）
   - PAY_0 的时间锚点和潜在泄漏分析
   - 敏感属性审计 + 差异影响分析 + 整改建议
   - 多阈值 precision-recall 权衡表
   - 概率校准作为 future work
   - 概念漂移和定期重训练建议
5. **论文标题建议**：不要写"我们成功预测了违约"，而是"信用风险评分模型的构建、评估与治理审计"
6. **不需要 rework**：模型方法合理，评估指标正确。主要问题在部署就绪度而非模型质量。
