# 设计短文：Solver 阶段对独立子任务 fan-out（多 agent 并行求解）

> 起因：Andy 在 iter3（穿越沙漠）Solver 卡死时提议——"既然 Solver 这么久，能不能再细分：一个负责分配任务、把解题拆给多个 agent 并行（也有先后顺序）？" 本文给出可行性结论与落地设计。供过目，**尚未实装**。

## 1. 一句话结论
**可行，且对"独立子任务"有真实收益（更快 + 顺带提升对拍独立性=纯净度）；但它治不了"单个算法爆炸"那种慢。** 用在刀刃上，不要无脑全切。

## 2. 哪些能并行（相互独立 → 真加速）
| 子任务 | 为何独立 | fan-out 收益 |
|---|---|---|
| **Q1 的 A、B 两条对拍范式** | A(时空网络+LP) 与 B(全状态DP) 本就是互证、互不依赖 | 拆给**两个互不通气的 agent** → 并行 ×2，**且独立性更强**：两实现真盲对拍比"同一 agent 自己跟自己对"更可信（正中纯净基准要害） |
| **Q3 敏感性扫描** | 每个参数点/每条单因子曲线独立 | embarrassingly parallel，N 个扫描点 N 路并行 |
| **多情景仿真 / 出图** | 各情景、各图独立 | 并行生成 |

## 3. 哪些不能（治不了 iter3 那次卡死）
- iter3 慢的根因是**算法爆炸**（一个 3.6 亿状态 DP 写在纯 Python 里），不是"串行任务堆太多"。**把一个巨型 DP 切给 10 个 agent，状态数不变，还是爆。**
- 真解法是**算法层**的（给 DP 设规模闸/维度解耦/粗化）——已沉淀为 `anti_patterns C5b` + Solver 角色 0b 性能闸。
- 教训：**先治算法可行性，再谈并行**。并行是"很多独立活儿串行太久"的解药，不是"坏算法"的解药。

## 4. 落地设计（映射到现有架构）
- **Router 本就是"任务分配器"**（判型→排哪些角色→定顺序）。本提议=在 **Solver 这一棒内部再 fan-out 一层**，不需要新发明协调者。
- **依赖 DAG**（Andy 说的"先后顺序"）：Modeler 选型 → [Solver-A ∥ Solver-B ∥ Solver-Q3扫描] 并行 → **merge 步**汇总 → 统一 `results.json`/`frozen_numbers.json` → Critic。其中 A∥B 有 barrier（都要齐了才能比对），Q3 扫描可流水。
- **谁来合并**：一个轻量 merge 角色（或 Solver 主体）负责把并行产物拼成单一 `results.json` + 保证 frozen 数字口径一致（**关键风险点**，见 §5）。
- **工具支持**：编排者本就能"一条消息并行起多个 agent"；更确定性的做法是用 **Workflow**（`parallel()` 跑 A/B/扫描、`pipeline()` 串后续），自带并发上限与结果聚合。

## 5. 成本 / 风险（诚实权衡）
- **Token 成本↑**：多 agent = 多上下文，比单 Solver 贵。值不值取决于子任务是否真独立、是否真耗时。
- **协调开销**：拆分→合并要写"统一 schema + frozen 一致性校验"，否则并行产物口径打架（A/B 用了不同单位/精度就对不上）。
- **纯净度**：A∥B 由互不通气 agent 跑 = **纯净度净增**（更独立的对拍）；这是除速度外的主要理由。
- **不是所有题都需要**：优化型有天然可拆的 A/B/扫描；但很多题 Solver 本就秒级（iter3 的 A 范式 0.4s），fan-out 纯属过度工程。**按题判断，默认单 Solver，重型/可拆才 fan-out。**

## 6. 建议
- **先做小**：下一道重型优化/图论题时，试点"A∥B 双 agent 盲对拍 + Q3 扫描并行"，用 Workflow 编排，量一次真实加速比与 token 成本，再决定是否纳入 Router 的标准调度。
- 与 `anti_patterns C5b`（性能闸）配套：**先保证每条子任务自身性能可行，再并行**。
