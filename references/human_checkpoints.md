# 人把舵卡点 human_checkpoints

> 供 `modelcrew-router` 编排时排程，由各对应 Agent 在产出末尾触发。
> ModelCrew 是**建模副驾**而非全自动黑箱：可复用的**流程**由 Agent 自动跑，但**关键判断**留 4 个显式卡点交给人。
> 这与本框架"硬核建模灵感由人把舵、Critic 诚实标注不确定性"的主线一致——不是少了自动化，而是**把人放在最该思考的 4 个决策点上**。

## 为什么要卡点（而不是一路自动跑到底）
建模比赛真正拉开档次的判断——**题读对没有、用哪个模型、错误结论怎么处置、定稿前补哪一维**——都带有
价值取舍和领域直觉，AI 给得出候选、给不出**拍板**。把这 4 处交给人，既挡住"读错题/选错模型"这类头号失分，
又让作品始终是"人主导 + AI 加速"，而非"AI 替你决定"。

## 统一卡点格式
每个卡点 Agent 在产出工件后，**停下**并输出一个醒目区块，**等用户回应再继续**：
```
🛑 CHECKPOINT · CP<n> <名称>
- 现在的判断：……（一句话给出 Agent 的当前结论）
- 需要你确认/补充：……（具体到要拍板的 1–3 件事）
- 不确认的风险：……（如果跳过这步、判断错了会怎样）
请回复：确认 / 修改（写明哪条怎么改）/ 补充
```
> 卡点是**软暂停**：用户可一句"确认继续"放行，也可改写后再放行。CP3 是**条件触发**（仅当 Critic 出现 ❌ 时）。

## 四个卡点

| 卡点 | 触发位置 | 谁来问 | 人拍板什么 | 为什么放这 |
|---|---|---|---|---|
| **CP1 审题确认** | Analyst 产出 `1_analysis.md` 后 | `modelcrew-analyst` | 题型判定对不对？子问题有没有漏？关键假设贴合现实吗？ | **读错题是头号失分点**，错在这里后面全白做 |
| **CP2 选模型** | Modeler 产出 `3_model.md` 后 | `modelcrew-modeler` | 在 2–3 个候选模型里选哪个；补领域直觉/创新点 | 模型选型是胜负手，且最吃人的领域经验 |
| **CP3 ❌处置**（条件） | Critic 产出 `5_audit.md` 且**有 ❌** 时 | `modelcrew-critic` | 被证伪的结论：如实写进局限性 vs 回炉重做该环节 | 处置错误结论是价值判断，关系到诚实与工作量的取舍 |
| **CP4 评委复盘** | Judge 产出 `7_review.md` 后 | `modelcrew-judge` | 看各维得分，决定补强哪一维 / 是否再迭代一轮 | 定稿前最后一次"花有限精力提最多分"的取舍 |

## 编排位置（嵌进 Router 时序）
```
Router 判型 → Analyst →[🛑CP1]→ (Scout) → Modeler →[🛑CP2]→ Solver
   → ⭐Critic →[🛑CP3 若有❌]→ Writer(含 LaTeX) → 评委Judge →[🛑CP4]→ 定稿
```

## 进阶：逼问模式（borrow `/grill-me`）
默认卡点是**一次性提问 + 等回应**。对高风险决策（尤以 CP1 审题、CP2 选模型为甚），可切换为**逼问模式**：
不止问一句，而是像 `/grill-me` 那样**顺着用户的回答继续追问**，直到把决策树的每个分支都问到底——
例如 CP2 用户说"选 A"，就追问"那 A 在假设 X 不成立时怎么办？B 的哪个优点你确定不要了？创新点准备怎么命名？"。
目的：把人脑子里**没说出口的隐含判断**逼出来，避免"随口确认"。逼问到双方对该决策**没有新信息可挖**为止，再放行。

## 铁律
- 卡点是**请人思考**，不是走过场：Agent 必须把"当前判断 + 风险"讲清，让人能在 1 分钟内做出有信息量的决定。
- 用户选择"修改"时，按其指示 **diff-only 精修**，不整段重来。
- 在非交互/批处理场景下，可由 Router 配置为"自动放行并记录待人复核"，但**默认开启卡点**。
